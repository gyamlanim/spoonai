# RAG Ablation Study

## Configuration Summary

| Config | Chunking Method | Reranking Method | Avg Latency |
|---|---|---|---|
| **word + llm** | Fixed-size 30-word windows, 5-word overlap | GPT-4o-mini selects relevant chunk indices | 0.7040s |
| **word + rule** | Fixed-size 30-word windows, 5-word overlap | IDF-weighted keyword matching + phrase bonus | 0.0003s |
| **sentence + llm** | Sentence-boundary, 2 sentences/chunk, 1-sentence overlap | GPT-4o-mini selects relevant chunk indices | 0.6309s |
| **sentence + rule** | Sentence-boundary, 2 sentences/chunk, 1-sentence overlap | IDF-weighted keyword matching + phrase bonus | 0.0003s |

---

## How Each Configuration Works

| Config | Chunking | Reranking | Benefits | Tradeoffs |
|---|---|---|---|---|
| **word + llm** | Sliding word windows; often cuts mid-sentence | LLM reasons about relevance even without keyword match | Semantically precise; handles paraphrase | Cuts sentence boundaries; high latency + API cost |
| **word + rule** | Same word-window chunking | Scores by rare keyword overlap; +5.0 bonus for exact phrase match | Extremely fast; deterministic; free | No semantic understanding; fails on conceptual queries |
| **sentence + llm** | Punctuation-boundary splits; grammatically complete pairs | LLM picks chunks that directly answer the query | Best overall quality; handles conceptual queries; overlap prevents boundary loss | Highest latency; API call per query |
| **sentence + rule** | Same sentence-aware chunking | Rare keyword overlap + exact phrase bonus; no LLM | Near-instant; clean readable chunks; best speed-to-quality for factual queries | Fails on semantic mismatch (e.g. "RAG in AI" vs "retrieval-augmented generation") |

## Key Takeaways

- **Sentence chunking strictly dominates word chunking** on output quality — `chunk_sentence_based` respects natural language boundaries, while word windows frequently cut mid-sentence and return incomplete or disjointed text.
- **LLM vs rule-based reranking is a latency/cost vs accuracy tradeoff** — rule-based is ~2000x faster (0.0003s vs 0.63–0.70s) but blind to semantics and fails when query phrasing does not match chunk vocabulary (e.g. "RAG in AI" vs chunks containing "retrieval-augmented generation").
- **Sentence + LLM is the highest-quality configuration** and is what Spoon uses in production (`app/services/rag.py`).
- **Sentence + rule is the best speed-to-quality ratio** for straightforward factual queries where keyword overlap is sufficient.
